## Вопрос 1

Что выведет этот код? Объясните порядок выполнения блоков.

```java
class Demo {
    static { System.out.print("A "); }
    { System.out.print("B "); }
    Demo() { System.out.print("C "); }
    public static void main(String[] args) {
        System.out.print("D ");
        new Demo();
        System.out.print("E ");
        new Demo();
    }
}
```

**Предсказанный вывод:**

```
A D B C E B C
```

**Объяснение:**

1. При загрузке класса выполняется статический блок — выводится A
2. Выполняется main:
- System.out.print("D ") — выводится D
- Создание первого объекта new Demo():
- Выполняется блок инициализации { System.out.print("B "); } — выводится B
- Выполняется конструктор Demo() { System.out.print("C "); } — выводится C
- System.out.print("E ") — выводится E
- Создание второго объекта new Demo():
- Снова выполняется блок инициализации (B) и конструктор (C) — вывод B C
Порядок выполнения: статический блок → main → блок инициализации → конструктор (при каждом создании объекта).

## Вопрос 2

Класс `Child` в пакете `b`, класс `Parent` в пакете `a`. `Child extends Parent`. Какие члены `Parent` доступны в `Child`? А какие доступны в другом классе пакета `b`, который НЕ наследует `Parent`?

**Ваш ответ:**

В Child (наследник):
- public — доступны
- protected — доступны (в другом пакете доступны только наследникам)
- package-private (без модификатора) — недоступны (другой пакет)
- private — недоступны

В другом классе пакета b, который не наследует Parent:
- public — доступны
- protected — недоступны (только для наследников)
- package-private — недоступны (другой пакет)
- private — недоступны

## Вопрос 3

Можно ли создать sealed-класс, у которого все наследники — record? Какое преимущество это даёт при использовании в switch?

**Ваш ответ:**

Да, можно. Sealed-класс или интерфейс может разрешать наследниками любые типы: классы, enum, record.
Преимущество в switch: компилятор знает все возможные подтипы, поэтому при использовании pattern matching
в switch не требуется default-ветка, если обработаны все разрешённые варианты.
Это даёт исчерпываемость (exhaustiveness) — если позже добавится новый наследник, компилятор укажет на незавершённый switch.

## Вопрос 4

Что выведет этот код? Объясните, что такое String Pool и как работает `intern()`.

```java
String a = "Hello";
String b = new String("Hello");
String c = b.intern();
System.out.println(a == b);
System.out.println(a == c);
System.out.println(b == c);
```

**Предсказанный вывод:**

```
false
true
false
```

**Объяснение:**

String Pool (пул строк): это специальная область памяти, где хранятся уникальные строковые литералы.
При создании строки через литерал ("Hello") она помещается в пул.

a = "Hello" — ссылка на строку в пуле
b = new String("Hello") — создаётся новый объект в куче (heap), не в пуле
c = b.intern() — если в пуле уже есть строка с таким содержимым ("Hello"), intern() возвращает ссылку на объект из пула (то есть на a)

## Вопрос 5

Почему этот код не компилируется? Как исправить?

```java
interface A { default void hello() { System.out.println("A"); } }
interface B { default void hello() { System.out.println("B"); } }
class C implements A, B { }
```

**Ваш ответ:**

Код не компилируется из-за конфликта имён методов по умолчанию (default methods). Класс C наследует два
одинаковых метода hello() от интерфейсов A и B, и компилятор не может определить, какой из них использовать.

class C implements A, B {
    @Override
    public void hello() {
        A.super.hello(); // или B.super.hello()
    }
}

## Вопрос 6

В чём разница между `StringBuilder` и `StringBuffer`? Приведите конкретный сценарий, когда нужен именно `StringBuffer`.

**Ваш ответ:**

Сравнение StringBuilder и StringBuffer
1. Потокобезопасность:
- StringBuilder: Нет
- StringBuffer: Да (синхронизированные методы)

2. Производительность:
- StringBuilder: Выше
- StringBuffer: Ниже из-за синхронизации

3. Появился в:
- StringBuilder: Java 5
- StringBuffer: Java 1.0

Сценарий для StringBuffer: когда строка модифицируется в многопоточной среде, и несколько потоков одновременно
обращаются к одному объекту (например, в веб-приложении или при логировании). Если нужна потокобезопасность,
выбирают StringBuffer. В однопоточном коде предпочтительнее StringBuilder — он быстрее.

## Вопрос 7

Что выведет этот код? Объясните, что происходит с переменной `x` в лямбде.

```java
int x = 10;
Runnable r = () -> System.out.println(x);
// x = 20;  // что будет, если раскомментировать?
r.run();
```

**Предсказанный вывод:**

```
10
```

**Объяснение:**

Лямбда-выражение может захватывать переменные из окружающего контекста, но только effectively final (эффективно финальные).
Переменная x не изменяется после инициализации, поэтому лямбда может её использовать.

Если раскомментировать x = 20;, код перестанет компилироваться, так как переменная перестанет быть effectively final.
Ошибка будет примерно: local variables referenced from a lambda expression must be final or effectively final.

## Вопрос 8

Напишите пример лямбда-выражения, которое нельзя заменить на ссылку на метод. Объясните почему.

**Ваш пример и объяснение:**

List<String> list = Arrays.asList("a", "bb", "ccc");
list.stream()
    .map(s -> s.length() > 2 ? s.toUpperCase() : s.toLowerCase())
    .forEach(System.out::println);

Лямбда содержит условную логику (тернарный оператор). Ссылка на метод может ссылаться только на готовый метод,
принимающий параметр и возвращающий значение. Если внутри лямбды есть логика, не укладывающаяся в вызов одного
существующего метода, её нельзя заменить ссылкой на метод. Можно только вынести логику в отдельный метод,
а затем использовать ссылку на него.

## Вопрос 9

Чем `EnumMap` быстрее `HashMap` для ключей-enum? Опишите внутреннюю реализацию в одном предложении.

**Ваш ответ:**

EnumMap использует массив фиксированного размера, где индексом выступает порядковый номер (ordinal) enum-константы,
что позволяет выполнять операции вставки, поиска и удаления за O(1) без вычисления хеш-кода и разрешения коллизий.

## Вопрос 10

Дан `record Point(int x, int y) {}`. Что сгенерирует компилятор автоматически? Можно ли добавить в record изменяемое поле? Можно ли наследоваться от record?

**Ваш ответ:**

1. Автоматически генерируется:
- Конструктор со всеми компонентами (canonical constructor)
- Геттеры (x(), y())
- equals() и hashCode()
- toString()

2. Можно ли добавить изменяемое поле?
Нет. Все компоненты record финальные (final) и неизменяемые. Record предназначен для неизменяемых данных (value objects).

3. Можно ли наследоваться от record?
Нет. Record является финальным (final) — от него нельзя наследоваться.
Это сделано для того, чтобы record оставался простым контейнером данных с предсказуемым поведением.
